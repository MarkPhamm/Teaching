1. Why dbt Exists

The problem it solves and how it changed analytics

01
1.1

The Problem Before dbt

What Teams Were Doing

  • Hundreds of SQL scripts in random folders
  • No version control — "final_v3_REAL_final.sql"
  • No testing — broken logic found by end users
  • No documentation — tribal knowledge only
  • No dependency management — run scripts in the right order... manually

What dbt Introduced

  • SQL files in a Git repo with clear structure
  • Version-controlled, code-reviewed transformations
  • Built-in testing framework for data quality
  • Auto-generated documentation and lineage
  • Automatic dependency resolution via ref()
In one sentence: dbt treats SQL transformations like software — with version control, testing, documentation, and CI/CD.
1.2

What dbt Actually Does

You write SELECT statements. dbt handles the rest.

Compile

Turns your SQL + Jinja templates into pure SQL that the warehouse understands

Build

Runs compiled SQL against the warehouse — creates tables and views in the right order

Test

Validates data quality after every run — not null, unique, accepted values, relationships

Document

Auto-generates a browsable site with model descriptions, column details, and lineage graphs

You never write CREATE TABLE or INSERT INTO. You write a SELECT, and dbt decides how to materialize it based on your configuration.
1.3

dbt Core vs dbt Cloud

What We Use

dbt Core (Open Source)

  • Free, open-source CLI tool
  • Run from your terminal or CI/CD pipeline
  • Full control over execution environment
  • Pairs with any orchestrator (Airflow, Dagster)

dbt Cloud (SaaS)

  • Managed service by dbt Labs
  • Web IDE, job scheduling, alerting
  • Built-in CI for pull requests
  • Costs money — free tier available
Same SQL, same project structure. Cloud just adds a managed layer on top of Core.

2. Project Structure

How a dbt project is organized

02
2.1

Anatomy of a dbt Project

my_project/
├── dbt_project.yml ← project config
├── models/ ← your SQL transformations
│   ├── staging/ ← 1:1 with sources
│   ├── intermediate/ ← business logic
│   └── marts/ ← final outputs
├── tests/ ← custom data tests
├── macros/ ← reusable SQL/Jinja
├── seeds/ ← CSV lookup tables
├── snapshots/ ← SCD Type 2
└── analyses/ ← ad-hoc queries

models/

The heart of the project. Each .sql file is a model — a SELECT statement that dbt materializes into a table or view.

YAML files

Live alongside models. Define sources, tests, descriptions, and column metadata.

macros/

Reusable Jinja functions — like utility functions for your SQL.

2.2

Key Configuration Files

dbt_project.yml

name: 'my_project'
version: '1.0.0'
profile: 'my_project'

models:
  my_project:
    staging:
      +materialized: view
    marts:
      +materialized: table

profiles.yml

# Lives in ~/.dbt/profiles.yml
# Defines warehouse connection

my_project:
  target: dev
  outputs:
    dev:
      type: duckdb
      path: dev.duckdb
profiles.yml is NOT in the project repo — it contains credentials. Each developer has their own.

3. Models & Layers

The building blocks of dbt transformations

03
3.1

What is a Model?

A model is a .sql file containing a single SELECT statement. dbt compiles it and creates a table or view in your warehouse.

What you write

-- models/staging/stg_orders.sql

SELECT
  id AS order_id,
  user_id,
  status,
  created_at
FROM {{ source('raw', 'orders') }}

What dbt generates

-- Compiled SQL

CREATE VIEW analytics.stg_orders AS (
  SELECT
    id AS order_id,
    user_id,
    status,
    created_at
  FROM raw.orders
);
You write the logic. dbt handles the DDL. No CREATE TABLE, no INSERT INTO, no DROP IF EXISTS.
3.2

The Three Model Layers

Staging

1:1 with raw tables
Clean & rename

→

Intermediate

Business logic
Joins & aggregations

→

Marts

Final outputs
Ready for consumption

Staging (stg_)

  • One model per source table
  • Light transformations only: rename, cast, reformat
  • Materialized as views
  • Never joins — just cleans a single source

Intermediate (int_)

  • Combine staging models
  • Apply business logic, joins, filters
  • Not exposed to end users
  • Break complex logic into steps

Marts (fct_, dim_)

  • Final tables consumed by BI tools
  • fct_ = facts (events, transactions)
  • dim_ = dimensions (entities, lookups)
  • Materialized as tables
3.3

Naming Conventions

LayerPrefixExamplePurpose
Stagingstg_stg_stripe__paymentsClean a single raw source
Intermediateint_int_payments_pivotedCombine and reshape
Factfct_fct_ordersEvents and transactions
Dimensiondim_dim_customersEntities and attributes
Double underscore in staging: stg_<source>__<table>
This tells you exactly where the data came from.
Anti-pattern: naming models after what they do instead of what they are.
transform_orders → stg_shopify__orders

4. Materializations

How dbt writes results to the warehouse

04
4.1

Materialization Types

View

A saved query — re-executes every time it's read. No storage cost, always fresh.

CREATE VIEW AS
SELECT ...

Table

Physically stored. Rebuilt from scratch on every run. Fast reads, slower builds.

CREATE TABLE AS
SELECT ...

Incremental

Only processes new/changed rows. Best for large, append-heavy tables.

INSERT INTO ...
WHERE new rows

Ephemeral

Not stored — injected as a CTE into downstream models. Like a reusable SQL snippet.

WITH model AS (
  SELECT ...
)
4.2

When to Use What

MaterializationUse WhenAvoid WhenTypical Layer
ViewSmall data, always need latest, stagingSlow queries, downstream depends on it heavilyStaging
TableEnd-user models, BI tools read from itData is huge and mostly append-onlyMarts
IncrementalLarge tables, event logs, append-heavySmall tables, complex business logicMarts (large)
EphemeralDRY reusable logic, not needed as outputNeed to query the model directlyIntermediate (sometimes)
Default strategy: staging = view, marts = table, upgrade to incremental only when table rebuilds become too slow.
4.3

Incremental Models — How They Work

{{
  config(
    materialized='incremental',
    unique_key='event_id'
  )
}}

SELECT
  event_id,
  user_id,
  event_type,
  created_at
FROM {{ source('raw', 'events') }}

{% if is_incremental() %}
  WHERE created_at > (
    SELECT MAX(created_at)
    FROM {{ this }}
  )
{% endif %}

First Run

Builds the full table from scratch (ignores the is_incremental() block).

Subsequent Runs

Only selects rows newer than the latest created_at already in the table.

unique_key

Handles upserts — if a row with the same key exists, it updates instead of duplicating.

Full refresh: run dbt run --full-refresh to rebuild from scratch when logic changes.

5. Sources & ref()

How dbt knows where data comes from and how models connect

05
5.1

Sources

Sources declare the raw tables that exist in your warehouse before dbt touches them.

_sources.yml

version: 2

sources:
  - name: stripe
    schema: raw_stripe
    tables:
      - name: payments
      - name: customers
        loaded_at_field: _etl_loaded_at
        freshness:
          warn_after:
            count: 12
            period: hour

Why Declare Sources?

  • Lineage — dbt tracks what raw tables your models depend on
  • Freshness — alert when source data is stale
  • Documentation — sources appear in the DAG and docs site
  • Abstraction — if a schema name changes, update one YAML file, not 50 models
Usage: {{ source('stripe', 'payments') }}
Resolves to the actual schema.table in the warehouse.
5.2

ref() — The Most Important Function

ref() references another dbt model. It's how models connect to each other.

-- models/marts/fct_orders.sql

SELECT
  o.order_id,
  o.customer_id,
  o.status,
  p.amount
FROM {{ ref('stg_shopify__orders') }} o
LEFT JOIN {{ ref('stg_stripe__payments') }} p
  ON o.payment_id = p.payment_id

What ref() Does

  • Resolves names — translates model name to actual schema.table
  • Builds the DAG — dbt knows fct_orders depends on both staging models
  • Execution order — staging runs first, marts run after
  • Environment aware — resolves to dev or prod schema automatically
Never hardcode table names. Always use ref() for dbt models and source() for raw tables. Hardcoded names break lineage and environment switching.

6. Testing &
Documentation

Trust your data — prove it's correct

06
6.1

Built-in Tests (Generic Tests)

Tests are assertions about your data. dbt runs them after building models and fails if any row violates the rule.

_schema.yml

models:
  - name: stg_orders
    columns:
      - name: order_id
        tests:
          - not_null
          - unique
      - name: status
        tests:
          - accepted_values:
              values: ['placed', 'shipped', 'completed']
      - name: customer_id
        tests:
          - relationships:
              to: ref('stg_customers')
              field: customer_id

not_null

Asserts no NULL values in the column

unique

Asserts every value is distinct — validates your primary key

accepted_values

Column only contains values from your allowlist

relationships

Foreign key check — every value exists in the referenced model

6.2

Custom Tests (Singular Tests)

Write any SQL query as a test. If it returns rows, the test fails.

tests/assert_no_negative_amounts.sql

-- Returns rows where amount is negative
-- If any rows come back, the test FAILS

SELECT
  order_id,
  amount
FROM {{ ref('fct_orders') }}
WHERE amount < 0

When to Use Custom Tests

  • Business rules that go beyond the 4 generic tests
  • Cross-model validation (totals should match)
  • Detecting specific data anomalies
  • Revenue should never be negative
Running tests:
dbt test — run all tests
dbt test --select stg_orders — test one model
dbt build — run models + tests together
6.3

Documentation

Add Descriptions in YAML

models:
  - name: fct_orders
    description: "One row per order. Includes
      payment amount and customer info."
    columns:
      - name: order_id
        description: "Primary key"
      - name: amount
        description: "Payment in USD"
dbt docs generate → builds the site
dbt docs serve → opens it in your browser

What You Get

Auto-generated docs site

  • Model descriptions & column details
  • Interactive lineage graph (DAG)
  • Source freshness status
  • Test coverage overview
Best practice: documentation lives next to the code in YAML. When the model changes, update the docs in the same PR.

7. Jinja, Macros
& Packages

Power features for DRY, reusable SQL

07
7.1

Jinja Templating

dbt uses Jinja — a Python templating language — inside SQL files. It lets you add logic, loops, and variables to your SQL.

Expressions {{ }}

-- Output a value
{{ ref('stg_orders') }}
{{ var('start_date') }}

Outputs the result of a function or variable.

Statements {% %}

-- Control flow
{% if target.name == 'dev' %}
  LIMIT 100
{% endif %}

Logic: if/else, for loops, set variables.

Comments {# #}

{# This won't appear
   in compiled SQL #}

Jinja comments — stripped during compilation.

Don't overdo it. Jinja makes SQL harder to read. Use it for DRY patterns, not to build a programming language inside SQL.
7.2

Macros — Reusable SQL Functions

macros/cents_to_dollars.sql

{% macro cents_to_dollars(column_name) %}
  ({{ column_name }} / 100.0)
{% endmacro %}

Using it in a model

SELECT
  order_id,
  {{ cents_to_dollars('amount_cents') }}
    AS amount_dollars
FROM {{ ref('stg_payments') }}

When to Use Macros

  • Same logic repeated across many models
  • Unit conversions (cents → dollars, UTC → local)
  • Standard cleaning patterns (trim, lower, coalesce)
  • Generating SQL dynamically (pivot columns, union sources)

When NOT to Use Macros

  • Logic used only once — just write the SQL inline
  • Making SQL "clever" at the cost of readability
  • A package already provides the macro
7.3

Packages — Community Macros

Packages are shared dbt projects with pre-built macros, tests, and models you can install.

packages.yml

packages:
  - package: dbt-labs/dbt_utils
    version: [">=1.0.0", "<2.0.0"]
  - package: dbt-labs/codegen
    version: [">=0.12.0", "<1.0.0"]
Install with: dbt deps

dbt_utils (Essential)

  • surrogate_key — generate hash-based surrogate keys
  • pivot — pivot rows to columns
  • union_relations — union multiple tables dynamically
  • date_spine — generate date dimension tables

codegen

Auto-generates staging models and YAML schemas from your sources. Saves hours of boilerplate.

dbt Best Practices

Do

  • One model, one purpose — don't cram multiple transforms into one file
  • Always use ref() and source() — never hardcode table names
  • Test your primary keys — unique + not_null on every model
  • Follow the layer pattern — staging → intermediate → marts
  • Document as you build — descriptions in the same PR as the model
  • Use version control — every change goes through a PR with review

Don't

  • Don't skip staging — marts that read directly from raw tables are fragile
  • Don't over-use Jinja — if the SQL is unreadable, you've gone too far
  • Don't make everything incremental — only when table rebuilds are too slow
  • Don't build one giant model — break complex logic into intermediate steps
  • Don't ignore tests — untested models are liabilities
  • Don't commit profiles.yml — it has credentials

Key Takeaways

1

dbt = SQL + software engineering — version control, testing, docs, CI/CD for transformations

2

Three layers — staging (clean), intermediate (combine), marts (serve). Keep them separate.

3

ref() builds the DAG — dbt knows what depends on what and runs in the right order

4

Materializations matter — views for staging, tables for marts, incremental only when needed

5

Test everything — not_null + unique on every primary key, accepted_values on categorical columns

6

Documentation is free — add descriptions in YAML, dbt generates the rest automatically

7

Jinja & packages — keep SQL DRY, but don't sacrifice readability for cleverness
