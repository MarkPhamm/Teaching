Two Frameworks: ETL vs ELT

ETL

Extract → Transform → Load

Extract

→

Transform

→

Load

Transform data before loading into the warehouse. The traditional approach.

  • Transform happens on a separate server (Informatica, Talend, SSIS)
  • Only clean, structured data enters the warehouse
  • You lose the raw data — can't reprocess later

ELT

Extract → Load → Transform

Extract

→

Load

→

Transform

Load raw data first, transform inside the warehouse. The modern approach.

  • Warehouse handles the compute (Snowflake, BigQuery, DuckDB)
  • Raw data is preserved — reprocess anytime
  • Transform with SQL (dbt) — accessible to analysts

Why ELT Won

FactorETL (Traditional)ELT (Modern)
When it emerged1990s — storage was expensive2010s — cloud made storage cheap
Transform happensOn a separate server before loadingInside the warehouse after loading
Raw data preserved?No — only transformed data is storedYes — always keep the original
ReprocessingMust re-extract from sourceJust re-run the transformation
ToolsInformatica, SSIS, Talenddbt, dlthub, Airbyte + Snowflake
Who transformsETL developers (Java, proprietary)Analytics engineers (SQL)
ScalabilityLimited by ETL server capacityScales with warehouse compute
Why ELT exists: Cloud warehouses made storage cheap and compute elastic. It no longer makes sense to transform before loading — load everything raw, preserve it, and let the warehouse do the heavy lifting.

1. Ingestion

Getting data from sources into your system

01
1.1

What is Ingestion?

The process of extracting data from various sources and loading it into your data warehouse.

OLTP Databases

PostgreSQL, MySQL, SQL Server — the databases behind your applications

APIs

REST/GraphQL from SaaS tools — Stripe, Salesforce, Google Analytics

Third-Party

Vendor data feeds, partner sharing, public datasets

Web Scraping

Last resort when no API exists — check ToS first

Key principle: Extract data as-is first. Never transform during ingestion — you lose the original.
1.2

Ingestion Tools

Primary
dlthub

Python-native, open-source data ingestion library. Lightweight, easy to get started, great for local development.

Alternatives

Airbyte
Fivetran
Stitch

Managed connectors with pre-built integrations. Less code, more configuration.

Extraction Methods

  • Full load — dump entire table
  • Incremental — only new/changed rows
  • CDC — stream every change in real time

3. Data Warehouse

Where your data lives for analytics

03
3.1

OLTP vs OLAP

OLTP (Transactional)

PostgreSQL
SQL Server
  • Optimized for writes
  • Powers your application
  • Row-based storage
  • Not designed for analytical queries

OLAP (Analytical)

Snowflake
BigQuery
  • Optimized for reads
  • Powers your analytics
  • Columnar storage
  • Fast aggregations, joins, window functions
Why separate? Querying OLTP directly for analytics would slow down the application. Extract → load into OLAP → analyze there.
3.2

Data Warehouse Tools

DuckDB
Snowflake
Databricks
BigQuery
Redshift

Pros

  • SQL-native, familiar to analysts
  • Managed — less operational burden
  • Built-in security, governance, scaling

Cons

  • Vendor lock-in (cloud warehouses)
  • Cost scales with compute usage
  • Less flexible for non-SQL workloads
Why DuckDB as primary? Free, runs locally, zero setup — perfect for learning and local ELT development. Production → Snowflake/BigQuery.
3.3

Lakehouse (Alternative)

An open architecture combining data lake flexibility with warehouse structure.

Object Storage

S3 — stores the actual data files

Table Format

Iceberg / Delta Lake — ACID, time travel

Query Engine

Trino / Spark — SQL over files

Catalog

Postgres / Hive / Glue — metadata

Open formats, full control, but more operational overhead. Best for engineering-heavy teams.
3.4

Warehouse vs Lakehouse

FactorWarehouseLakehouse
Team skillSQL-heavy, analyst-drivenEngineering-heavy
Data typesStructuredStructured + unstructured
Control needsLow — prefer managedHigh — need open formats
Budget modelPay per query / computePay for infra you manage
Speed to valueFaster to set upMore upfront investment
There is no universally "better" choice — it depends on the team, data, and business.

4. Transformation

Turning raw data into analytics-ready models

04
4.1

Why Transform?

Raw data is messy. Transformation cleans, structures, and models it for analytics.

Clean

Handle nulls, duplicates, formatting issues — make data trustworthy

Model

Build dimensional models (facts & dimensions), create business logic

Test

Validate data quality — not null, unique, accepted values, relationships

ELT pattern: Extract → Load raw into warehouse → Transform inside the warehouse using SQL. The warehouse handles the compute.
4.2

Transformation Tools

Primary
dbt core
  • SQL-based transformations
  • Version-controlled models
  • Built-in testing & documentation
  • Jinja templating for DRY SQL
  • Industry standard for analytics engineering

Alternative

sqlmesh

  • Newer, built on learnings from dbt
  • Built-in column-level lineage
  • Virtual environments for testing
  • Incremental by default

6. Orchestration

The brain of the pipeline

06
6.1

What Orchestration Does

Schedule

Trigger pipeline tasks on time or on events

Dependencies

Task B waits for task A — no race conditions

Retries

Handle failures gracefully with alerts and retries

Visibility

Logs, run history, data lineage

Without orchestration: you're running cron jobs and praying.
6.2

Orchestration Tools

Primary
Airflow

Industry standard, DAG-based, Python-native. Massive community, battle-tested in production at scale.

Prefect

Flow-based, Pythonic, less boilerplate. Great for getting started — just decorate your Python functions.

Dagster

Asset-based — define what you produce, not how to run it. Modern dev experience.

Key trend: DAG-based (define task order) → asset-based (define desired outputs).

Who Owns What?

Different roles own different parts of the pipeline. Think of it as backend vs frontend of the data team.

"Backend" — Feed data into the warehouse

  • Data Engineer (DE) — owns ingestion, staging, warehouse infra, orchestration, CI/CD. Builds and maintains the pipes.
  • Analytics Engineer (AE) — owns the transformation layer (dbt models), data quality tests, documentation. Bridge between DE and analysts.

"Frontend" — Consume data from the warehouse

  • Data Analyst (DA) — owns dashboards, reports, ad-hoc analysis. Translates data into business decisions.
  • Data Scientist (DS) — owns ML models, experiments, statistical analysis. Builds predictive features on top of clean data.

MLOps

Model deployment, monitoring, feature stores. Productionizes what DS builds.

Platform / Infra

Cloud infra, Kubernetes, networking, IAM. Keeps the whole thing running.

Data Governance

Access control, PII, compliance, data catalog. Makes sure data is used responsibly.

Key Takeaways

1

Ingestion — extract data as-is, never transform during extraction

2

Staging — always land raw data first. Preserve, decouple, enable replay.

3

Warehouse — start with DuckDB locally, scale to Snowflake/BigQuery in production

4

Transformation — dbt is the industry standard, SQL-based, testable, version-controlled

5

BI Tools — BI as Code prevents dashboard sprawl and metric inconsistency

6

Orchestration — without it, pipelines are just scripts. Airflow is the industry standard.

7

CI/CD — treat pipelines like software. GitHub Actions + Docker + Terraform.

These five components + CI/CD provide the optimal foundation. The knowledge transfers across the entire data engineering landscape.
